# Phase 2 実験結果まとめ (2026-04-17〜18)

## モデル構成

- SimpleGPT2: 31.9M params (hidden 512, 8 layers, 8 heads, vocab ~6500)
- 文字単位トークナイザ (入力ひらがな ~180, 出力漢字かな混じり ~6500)
- Causal LM, teacher forcing, fp16 混合精度, AdamW + cosine decay

## データバージョンの遍歴

| Version | MeCab feature | 問題 | 結果 |
|---------|--------------|------|------|
| v1 | features[7] (書字形基本形) | 漢字が reading に混入 | postprocess で 99.5% 除去 |
| v2 | features[9] (発音形出現形) | 長音化 (よほう→よほー), 助詞変化 (は→わ) | 短文コピー問題の主因 |
| v3 | **features[17] (仮名形出現形)** | なし | 正解。活用形も正しい |

### unidic-lite feature レイアウト (重要)

```
[6]  書字形出現形    辞書形カタカナ (スル, ダ) — 活用語で NG
[7]  書字形基本形    漢字のまま返る — 読みとして使えない
[8]  出現形       surface そのまま — 漢字語で読みにならない
[9]  発音形出現形    音声学的 (ー, わ, お) — IME 入力と不一致
[17] 仮名形出現形    活用後の正しいカタカナ — 正解
```

## 手動テストの失敗パターン分類

### カテゴリ A: 短文コピー問題
- 「つかれた」→つかれた、「さむいふゆ」→さむいふゆ
- 31.9M モデルの容量限界 (データ量増やしても改善しない)
- 200M モデルで解消見込み

### カテゴリ B: 同音異義語
- 「かんじへんかん」→感じ変換 (漢字変換が正解)
- データ頻度通りの振る舞い (感じ > 漢字)
- 文脈モデリング強化で改善可能

### カテゴリ C: AR 構造崩壊
- 先頭の誤りが後続に波及
- CTC-NAT (並列生成) で構造的に解消

### カテゴリ D: 文字単位 AR の局所最適
- 「にほん→二本」(日本が正解、データ上 112倍多い)
- 文字単位で「に→二」が局所的に選ばれる
- CTC-NAT の並列生成で解消見込み

## Beam Search の結果

step 70000, beam=10, length_penalty=0.6, repetition_penalty=1.2:

| 指標 | 結果 |
|------|------|
| Greedy correct | 12/20 |
| Beam top1 correct | 11/20 |
| **In beam (any position)** | **19/20 (95%)** |

**モデルは正解を知っている (beam 内に 95% 含む) が、ランキングで top1 に出せない。**
→ KenLM shallow fusion やユーザ辞書でリランキングすれば 95% に到達可能。

## CPU 推論速度 (31.9M, fp32)

| 入力長 | 出力トークン | 合計時間 | ms/token |
|--------|------------|---------|----------|
| 短文 5文字 | 2 | 25ms | 12.4 |
| 中文 15文字 | 9 | 99ms | 11.0 |
| 長文 30文字 | 20 | 284ms | 14.2 |

AR は出力長に線形。CTC-NAT なら 1パスで 10-60ms (入力長依存のみ)。

## モデル容量 vs データ量

| データ量 | 天井 CharAcc | 手動テスト |
|---------|-------------|----------|
| 200万 (ローカル) | ~87-88% | - |
| 2012万 (vast.ai) | ~91.5% | 80/100 |
| **差分** | **+3-4%** | - |

2012万のうち 200万で 87% に飽和 → 残り 1812万で +4%。
31.9M モデルの天井は ~91-92%。これ以上は 200M モデルが必要。
