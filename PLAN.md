# new-ime: NAT ベース並列生成 日本語IME 開発計画

## 前提の修正と設計方針

### 学習時間の見積もり

- ACT 想定規模 (数十デモ、数M パラメータ) なら数時間で学習可能
- **100M パラメータの encoder-decoder をスクラッチ学習する場合は数時間では収まらない**
  - A100 単発: 数日
  - H100: 1〜2 日
- 日本語事前学習重み (cl-tohoku/bert-japanese 等) からエンコーダを初期化し、デコーダのみスクラッチ学習すれば十数時間レベルに短縮可能

### 「学習法の再発明」の実態

ACT の完全移植ではなく、**NAT (Non-Autoregressive Translation)** の既存研究が直接流用できる領域がかなりある。

Gu et al. 2018 以降、並列出力トランスフォーマーには以下のレシピが蓄積されている:

| 手法 | 概要 |
|------|------|
| Length prediction head | 出力長を事前予測 |
| Iterative refinement (Mask-Predict, CMLM) | 低信頼トークンをマスクして再予測 |
| Glancing Language Model (GLAT) | easy→hard カリキュラムで学習安定化 |
| Directed Acyclic Transformer (DAT) | DAG 構造で複数候補を並列探索 |

「短系列から短系列への離散並列生成」は NAT の応用に近い。ACT の CVAE 的要素はユーザ適応向けに後から足す二段構成が効率的。

**再発明が必要な部分**: アーキテクチャの組み合わせ方と IME タスク固有の制約の組み込み。基礎学習法は既存資産で回せる。

### 設計判断: NAT を選ぶ理由 (2026-04-17 確定)

zenz-v1 と同じ GPT-2 アーキテクチャでデータ品質のみで勝負する路線は採らない。差分はアーキテクチャで出す。

**NAT の構造的優位性 (調査結果)**:
- かな漢字変換は**単調アラインメント** (入出力の語順がほぼ同一) → NAT が最も得意とする構造
- GLAT + KD で AR との精度差は BLEU ~0.5 まで縮まる (MT ベンチマーク)
- DAT (Huang 2022) は KD なしで AR と同等以上を達成した実績あり
- 主な精度リスクは同音異義語の曖昧性解消 → CMLM refinement で緩和可能

**既知のトレードオフ**:
- llama.cpp/GGUF エコシステムは使えない (decoder-only 前提) → 推論基盤を自前で用意する必要がある (ONNX Runtime or ggml 直接)
- Zenzai は既にラティス→ニューラル検証のドラフト・検証ループを持っており、90M params の AR は CPU で ~30-75ms → IME 要件は AR でも満たせる
- つまり NAT のレイテンシ優位だけでは動機が弱い。**アーキテクチャの新規性と将来の CVAE 統合**が本質的な差分

**撤退経路**: Phase 3 Go/No-Go で AR+投機的デコードに切り替え可能。この場合でもデータパイプラインと評価基盤は再利用できる。

### 設計更新: CMLM → CTC-NAT (2026-04-17 調査後)

追加調査により、CMLM ではなく **CTC ベースの NAT** を主軸に変更。

- かな→漢字は**単調・圧縮的** (入力のかな長 >= 出力の漢字混じり長) → CTC の前提と完全一致
- CTC は**長さ予測を不要にする** (blank トークンで長さ差を吸収)
- CTC beam search で**複数候補を自然に生成**できる
- ASR の NAT (Mask-CTC, Paraformer) が先行事例
- fairseq は **2026-03 にアーカイブ済み** → スクラッチ実装 (PyTorch) が最善手
- 具体的実装計画は **ROADMAP.md** に記載

### 参考実装

| 実装 | URL | 備考 |
|------|-----|------|
| fairseq CMLM (参考のみ) | `facebookresearch/fairseq` | アーカイブ済み、コード参照のみ |
| DA-Transformer | `thu-coai/DA-Transformer` | ICML 2022、撤退先 (DAT) の参考 |
| NMLA-NAT | `ictnlp/nmla-nat` | CTC-NAT 実装、最も関連が近い |

### zenz-v1 アーキテクチャ詳細 (コードリーディング結果)

- **モデル**: GPT-2, 90M params, hidden 768, 12 層, 12 ヘッド, vocab 6000
- **ベース**: `ku-nlp/gpt2-small-japanese-char` からファインチューン
- **推論**: llama.cpp (GGUF Q8_0, ~97MB), 純粋 argmax デコーディング
- **パイプライン**: ラティスベース N-best 生成 → GPT-2 でスコアリング → 棄却なら制約修正して再ドラフト (最大 10 回)
- **プロンプト形式**: `[inputTag]カタカナ[contextTag]左文脈(40文字)[outputTag]` (特殊 Unicode 制御文字)
- **トークナイザ**: 文字単位 + バイト BPE, swift-tokenizers 経由
- **ライセンス**: CC-BY-SA 4.0

## Phase 1: データパイプライン

**期間**: 4〜6 週間

### 目的

学習データと評価データを用意する。ここを雑にやると後のすべてが腐る。

### データソース

| ソース | 特徴 |
|--------|------|
| Wikipedia JA dump (最新) | 大規模、正書法準拠 |
| 青空文庫 | 文学、文体多様性 |
| CC-100 Japanese | 大量だがノイズあり、フィルタ前提 |
| 公開コーパス (wikibooks、日本語ニュース記事等) | 口語寄りの補強 |

### 読み付与パイプライン

1. MeCab + ipadic-NEologd でトークン化・読み付与 (zenz-v1 と同じ資産)
2. UniDic 併用も検討 (ipadic-NEologd より精度が高い場面がある)
3. フィルタ:
   - 読み付与信頼度が低い文を除外
   - 外国語混入文の除外
   - 極端に短い/長い文の除外
4. **(かな読み, 漢字表記, 前文脈)** の三つ組を作る

### データ量見積もり

最初は **1000 万文ペア程度**で開始。スケールアップは性能を見てから。大きすぎる初期データは学習イテレーション速度を下げて設計の反復を遅らせる。

### 評価セット構築

- 学習データと分離した領域 (ドメイン、時期、ソース) から作成
- 分野別 (技術、文学、日常、専門) で層化サンプリング
- ユーザ適応評価用に、特定の書き手の文章セットも別途用意 (Phase 4 以降で使用)

### トークナイザ

| 側 | 方式 | 語彙数 |
|----|------|--------|
| 入力 (かな) | 文字単位 | 小 (ひらがな・カタカナのみ) |
| 出力 (漢字かな混じり) | SentencePiece BPE | 32k 程度 |

入出力は**独立語彙**を推奨。入力はひらがな・カタカナのみで語彙が小さく効率的。

### 成果物

- クリーンな学習/検証/テストセット
- tokenizer のモデルファイル
- データ統計レポート

### Go/No-Go 判定

検証セットで **mozc 単体のベースライン精度**を測る。これより悪いならデータ品質に問題あり、先に進まない。

## Phase 2: ベースライン実装

**期間**: 3〜4 週間

### 目的

データとインフラの正しさを、単純なモデルで検証する。新規性はここにはない。**正しく学習できることの確認**が目的。

### 構成

- GPT-2 スタイルの decoder-only モデル、60〜90M パラメータ (zenz-v1 相当)
- 入力: `前文脈 [SEP] かな読み` → 出力: `漢字表記`
- 学習: 通常の causal LM、teacher forcing
- fp16/bf16 混合精度、AdamW、warmup + cosine decay

### インフラ

| 項目 | 選択 |
|------|------|
| フレームワーク | PyTorch + HuggingFace Transformers |
| 実験管理 | Weights & Biases |
| GPU | A100 1 枚 (RunPod / Lambda / Vast.ai, $1〜3/hr) |
| 推定学習時間 | 10〜30 時間 |
| 総予算 | $50〜100 |

### 期待される精度

zenz-v1 には届かなくてもよい。**mozc を超えていればデータと学習コードは正しい**。

### 成果物

- 動くベースラインモデル
- 学習/評価スクリプト
- 精度レポート

### Go/No-Go 判定

mozc + UT を文字精度で明確に超えているか。超えていないなら Phase 3 に進む前にデータまたは学習設定に問題。

## Phase 3: 並列生成アーキテクチャ (本体)

**期間**: 6〜8 週間

### 目的

Encoder-decoder + 並列出力の本命アーキテクチャを組み上げる。

### アーキテクチャ

```
[前文脈 + かな入力]
        │
   ┌────▼────┐
   │ Encoder  │  ← 日本語 BERT 系から初期化, 6〜8 層, hidden 512
   └────┬────┘
        │
   ┌────▼──────────┐
   │ Length Predict │  ← 出力トークン長をクラス分類 (かな長の ±20%)
   └────┬──────────┘
        │
   ┌────▼────┐
   │ Decoder  │  ← 並列に N 個の出力トークンを生成, 6〜8 層
   │ (NAT)   │     cross-attention でエンコーダ出力を参照
   │          │     query tokens 方式 (ACT/DETR 系)
   └────┬────┘
        │
   ┌────▼────────┐
   │ Refinement   │  ← CMLM 的に低信頼トークンをマスクして再予測 (1 回パス)
   └─────────────┘
        │
   漢字かな混じり出力
```

合計パラメータ: **100M 前後**

### 学習設計

| 損失 | 内容 |
|------|------|
| 主損失 | cross-entropy on parallel output |
| 補助損失 1 | 長さ予測の CE |
| 補助損失 2 | エンコーダ側の span-level 対応予測 (読みと表記の alignment 監督、必要に応じて) |

### 学習の注意点

- NAT は自己回帰より学習が不安定になりがち → **GLAT 的な glancing sampling (easy→hard カリキュラム) を標準で入れる**
- Knowledge distillation (自己回帰モデル出力を教師にする) も NAT 常套手段 → **Phase 2 のベースラインを教師として使える**
- ACT 的要素として最初に入れるのは **query token 構造のみ**。CVAE は入れない (Phase 6 以降)

### 計算コスト

| 項目 | 見積もり |
|------|----------|
| GPU | H100 |
| 推定学習時間 | 20〜40 時間 |
| 総予算 | $100〜300 |

### 成果物

- 並列生成モデル
- 推論コード
- ベースライン比較レポート

### Go/No-Go 判定

ベースライン (Phase 2) と同等以上の精度を、**推論レイテンシ数分の 1 で**達成できているか。片方のみでは不十分。両立できなければ設計を見直すか、並列化の優位性自体を疑う。

### 失敗シナリオと撤退オプション

ここで詰まる可能性は十分ある。NAT の知見を動員しても並列生成は自己回帰より精度が落ちる傾向が経験上あり、IME ではその許容度が低い。

**撤退オプション**: 自己回帰 + 投機的デコード (speculative decoding) に舵を切る。これでもレイテンシ改善は実現できる。

## Phase 6: 量子化とユーザ適応 (v1.0 リリース後)

v1.0 で動作確認してから取り組む拡張。別トラック。

### 1.58-bit QAT

- Phase 3 で作った fp16 モデルを教師に、1.58-bit で QAT 再学習
- BitNet.cpp または自前の CPU 推論経路を整備
- **失敗時の撤退**: int8 または int4 で妥協

### CVAE 的ユーザ適応

- 潜在変数 z をユーザ ID または書き手 embedding として導入
- Phase 1 で用意した「特定書き手の文章セット」で少量ファインチューニング
- 推論時に z を固定、あるいはオンライン更新
- ユーザ辞書の統計を z に反映するメカニズム

## 総括

### スケジュール

| 条件 | 期間 |
|------|------|
| 週 10〜20 時間コミット | 6〜9 ヶ月 |
| フルタイム | 3〜4 ヶ月に圧縮可能 |

### 金銭コスト

GPU 借用で合計 **$500〜1500** 程度。100M〜300M クラスのタスクはクラウドで十分。

### リスクと機会

| | 内容 |
|---|------|
| **最大リスク** | Phase 3 の並列生成が精度面で詰まること。撤退経路 (自己回帰 + 投機的デコード) を用意済み |
| **最大機会** | Phase 3 が成功すれば IME に特化した軽量並列生成モデルという明確な研究貢献。論文化の余地あり。OSS リリースすれば Linux 日本語入力エコシステムに直接効く |

### 一貫性チェック

| 変更点 | 理由 |
|--------|------|
| 工数見積もり: 3〜5 ヶ月 → 6〜9 ヶ月 | 並列生成アーキテクチャの研究性を明示的にスコープに含めたため、Phase 3 の反復コストが現実的に。スコープの違いによる変動であり矛盾ではない |
| 設計源泉: ACT → NAT | zenz-v1 が貪欲で動く事実を踏まえ、並列化の動機をレイテンシに再定位した結果 |
| 学習時間: 数時間 → 数日 | 100M パラメータのスクラッチ学習の実態に合わせて訂正 |
| 維持された設計軸 | encoder-decoder 構造、小型モデル、1.58-bit、CPU 推論 — 一貫 |
